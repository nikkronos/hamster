# Безопасность

## Секреты и конфиденциальные данные

### Правила хранения секретов

- **НИКОГДА** не хардкодить пароли, токены, API-ключи в коде
- **НИКОГДА** не писать секреты в чате с агентом (даже в примерах)
- **НИКОГДА** не добавлять секреты в логи
- Все секреты хранить в `.env` файле (который в `.gitignore`)
- Файл `.env` всегда должен быть в `.gitignore`

### Переменные окружения

Все секреты загружаются через `data/config.py`:

```python
from environs import Env

env = Env()
env.read_env()

TOKEN = env.str("TOKEN")
ADMINS = [int(i) for i in env.list("ADMINS")]
API_ID = env.int("API_ID")
API_HASH = env.str("API_HASH")
BANK_CARD = env.str("BANK_CARD")
```

### Пример `.env` файла

```
TOKEN=your_bot_token_here
DEVELOPERS=123456789,987654321
ADMINS=123456789
API_ID=your_api_id
API_HASH=your_api_hash
BANK_CARD=your_card_number
```

## SQL-инъекции

### Критические правила

- **НИКОГДА** не использовать f-строки в SQL-запросах с пользовательскими данными
- Всегда использовать параметризованные запросы: `cursor.execute(query, params)`
- Для динамических имен столбцов использовать валидацию и whitelist

### Примеры

```python
# ПРАВИЛЬНО - параметризованный запрос
query = "SELECT * FROM users WHERE user_id = ?"
params = [user_id]
await execute_query(query, params)

# НЕПРАВИЛЬНО - SQL-инъекция!
query = f"SELECT * FROM users WHERE user_id = {user_id}"
await execute_query(query)
```

### Проблемные места

В текущем коде есть проблемные места:
- `handlers/admins/add_channel.py` - динамическое создание столбцов через f-строки
- `handlers/admins/delete_channel.py` - динамическое удаление столбцов через f-строки

**Решение:** Использовать валидацию channel_id и параметризованные запросы где возможно.

## Валидация входных данных

### Правила

- Все данные от пользователей должны валидироваться перед записью в БД
- Проверять типы данных и диапазоны значений
- Использовать `validators.py` для валидации

### Что валидировать

- `user_id` - должен быть положительным integer
- `channel_id` - должен быть integer (может быть отрицательным для каналов)
- `days` - количество дней подписки, должно быть положительным integer
- `amount` - сумма платежа, должна быть положительным числом
- `promocode` - текст промокода, должен соответствовать формату

## Rate Limiting

### Защита от спама

- При массовых операциях (рассылки) использовать задержки между сообщениями
- Обрабатывать ошибки `RetryAfter` от Telegram API
- Лимит Telegram: ~20 сообщений в секунду для ботов

### Примеры

```python
# Правильная обработка RetryAfter
try:
    await bot.send_message(chat_id, text)
except RetryAfter as e:
    await asyncio.sleep(e.timeout)
    await bot.send_message(chat_id, text)
```

## Доступ к админ-функциям

### Проверка прав

Все админские функции должны проверять права доступа:

```python
from data import config

if message.from_user.id not in config.ADMINS:
    return
```

### Список админов

Список админов хранится в `.env` файле и загружается в `config.ADMINS`.

## Логирование и приватность

### Что НЕ логировать

- Токены ботов
- API ключи
- Номера карт
- Пароли
- Личные данные пользователей (по возможности)

### Что логировать

- Действия администратора (без секретов)
- Ошибки (без секретов)
- Общую статистику
- ID пользователей (для отладки)

## Защита от ошибок

### Обработка исключений

- Все критические операции оборачивать в try-except
- Логировать ошибки с контекстом
- Не прерывать работу бота при ошибках отдельных операций

### Примеры

```python
try:
    await bot.send_message(user_id, text)
except Exception as e:
    logger.error(f"Ошибка отправки сообщения пользователю {user_id}: {e}")
    # Бот продолжает работу
```
