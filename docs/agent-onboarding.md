# Онбординг для AI агента

Этот документ содержит всю необходимую информацию для начала работы с проектом PastuhiBot.

## Краткое описание проекта

**PastuhiBot** - Telegram-бот для управления подписками на доступ к копиям каналов.

### Основные функции:
- Управление подписками пользователей (1/3/6/12 месяцев)
- Автоматическое создание пригласительных ссылок для доступа к каналам
- Управление доступом: бан/разбан при истечении подписки
- Административная панель: управление каналами, подписками, промокодами
- Планировщик задач: автоматические напоминания об окончании подписки

### Целевая аудитория:
- Пользователи, желающие получить доступ к копиям каналов
- Администратор (управление каналами и подписками)

## Архитектура проекта

Подробная информация об архитектуре: [docs/architecture.md](architecture.md)

### Основные компоненты:

- **`bot.py`** - точка входа, инициализация БД, запуск планировщика и polling
- **`data/config.py`** - загрузка переменных окружения из `.env`
- **`loader.py`** - создание объектов `bot`, `dp`, `scheduler`
- **`database/`** - вся логика работы с SQLite:
  - `keys.py` - структура таблиц
  - `work_db.py` - подключение и выполнение запросов
  - `getters.py` - функции получения данных
  - `setters.py` - функции изменения данных
- **`utils/`** - утилиты:
  - `misc.py` - бизнес-логика (подписки, ссылки, бан/разбан)
  - `states_users.py` - работа с FSM состояниями
- **`handlers/`** - обработчики сообщений:
  - `admins/` - админские функции
  - `author/` - функции автора
  - `start.py`, `pay.py`, `about_subscribe.py` и др. - пользовательские функции
- **`keyboards/`** - клавиатуры для интерфейса

### База данных:
- **SQLite** (файл `database/database.sqlite`)
- Таблицы: `users`, `tracked_channels`, `user_subscriptions`, `user_links`, `payments`, `promocodes`, `tasks_for_scheduler`

## Бизнес-правила

Подробная информация: [docs/business-rules.md](business-rules.md)

### Ключевые правила:

1. **Подписка и доступ:**
   - Доступ к каналам только для пользователей с активной подпиской
   - Срок подписки считается до конкретной даты и времени (`datetime_end_subscribe`)

2. **Продление и оплата:**
   - Оплата подтверждается вручную администратором по скриншоту
   - При подтверждении оплаты подписка продлевается на указанное количество дней

3. **Уведомления и исключение:**
   - Бот проверяет подписки через планировщик
   - 3 дня до конца: напоминание
   - 1 день до конца: напоминание
   - Срок истек: статус → удаление подписки, бан из каналов, разбан для возможности вернуться

4. **Каналы:**
   - `channel_id` - ID канала-источника (откуда копировать)
   - `forward_channel_id` - ID канала-назначения (куда копировать)
   - Для каждого канала создаются пригласительные ссылки для пользователей

## Паттерны и стандарты кода

Подробная информация: [docs/patterns.md](patterns.md)

### Общие правила:

- Все настройки (токены, пароли) в `.env` через `data/config.py`
- SQLite через `sqlite3`, параметризованные запросы (`?`)
- Логирование через `logging`, не использовать `print()`
- Все вызовы API Telegram и БД оборачивать в `try-except`
- Использовать FSM для многошаговых диалогов

### Безопасность:
- **НИКОГДА** не хардкодить секреты в коде
- **НИКОГДА** не писать секреты в чате с агентом
- Все секреты в `.env` (в `.gitignore`)
- Использовать валидацию входных данных

### Логирование:
- Логировать все важные операции (действия админа, ошибки БД, ошибки API)
- Уровни: `DEBUG`, `INFO`, `WARNING`, `ERROR`, `CRITICAL`
- В логах не должно быть секретов

## Workflow работы агента

### Начало работы:

1. **Прочитать базу знаний:**
   - Этот документ (agent-onboarding.md)
   - Связанные документы (architecture.md, business-rules.md, patterns.md)

2. **Создать спек:**
   - Использовать шаблон `docs/specs/spec-template.md`
   - Описать задачу, риски, альтернативы
   - Создать чеклист для самопроверки

3. **Задать вопросы:**
   - Уточнить неясные моменты у пользователя
   - Проверить понимание требований

4. **Самооценка решения:**
   - Это лучшее решение? Почему?
   - Какие подводные камни?
   - Какие риски?
   - Альтернативные решения?

5. **Реализация:**
   - Реализовать с тестами
   - Проверить все связанные функции
   - Убедиться, что не сломались существующие функции

### Критически важно перед изменением:

- **Проверить все связанные функции** (использовать `grep`/`codebase_search`)
- **Убедиться, что все кнопки имеют обработчики**
- **Проверить навигацию между меню**
- **Проверить, что изменения не сломают существующие функции**

### Завершение работы:

Провести финальную проверку:
1. Чеклист из спека выполнен?
2. Проверены все связанные функции?
3. Все кнопки имеют обработчики?
4. Навигация между меню работает?
5. Все тесты написаны?
6. Соответствует patterns.md?
7. Логирование добавлено?
8. Нет hardcoded секретов?
9. Документация обновлена?
10. Линтер не находит ошибок?

**КРИТИЧЕСКИ ВАЖНО:** Перед коммитом убедиться, что не сломались существующие функции!

## Важные напоминания

1. **Всегда проверяй связанные функции** перед изменением кода
2. **Тестируй изменения** перед коммитом
3. **Не коммить сломанный код**
4. **Следуй паттернам** из `docs/patterns.md`
5. **Логируй важные операции**
6. **Никогда не хардкоди секреты**
