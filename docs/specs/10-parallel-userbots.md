# Техническое задание: Параллельные юзерботы для копирования каналов

## Что надо сделать

Реализовать систему параллельных юзерботов (Pyrogram) для копирования содержимого каналов, чтобы обеспечить отказоустойчивость и предотвратить остановку работы при блокировке одного из ботов.

## Как это должно работать

### Текущая ситуация

- Один юзербот копирует содержимое всех каналов
- При блокировке юзербота вся система копирования останавливается
- Это критическая проблема для работы бота

### Целевое состояние

- Несколько юзерботов работают параллельно
- Каждый юзербот отслеживает свой набор каналов
- При блокировке одного бота остальные продолжают работать
- Автоматическое перераспределение каналов при блокировке

### Сценарии использования

1. **Нормальная работа:**
   - Каждый юзербот копирует свои каналы
   - Все каналы обрабатываются параллельно

2. **Блокировка одного бота:**
   - Обнаружение блокировки (ошибки при отправке сообщений)
   - Автоматическое перераспределение каналов на другие боты
   - Уведомление администратора

3. **Добавление нового канала:**
   - Выбор наименее загруженного бота
   - Назначение канала этому боту

4. **Добавление нового юзербота:**
   - Равномерное перераспределение каналов между всеми ботами

## Потенциальные проблемы и риски

- **Риск 1:** Блокировка всех юзерботов одновременно
  - **Митигация:** Использовать разные IP-адреса (VPN/прокси), разные номера телефонов, разные аккаунты

- **Риск 2:** Telegram может заблокировать при подозрительной активности
  - **Митигация:** Rate limiting, задержки между операциями, имитация человеческого поведения

- **Риск 3:** Сложность синхронизации между ботами
  - **Митигация:** Централизованная БД для хранения состояния, четкое распределение каналов

- **Риск 4:** Потеря данных при перераспределении
  - **Митигация:** Транзакции БД, проверка целостности данных

- **Риск 5:** Увеличение нагрузки на сервер
  - **Митигация:** Мониторинг ресурсов, оптимизация кода

## Альтернативные решения

- **Вариант А:** Один юзербот с несколькими сессиями
  - Минусы: При блокировке все сессии блокируются, сложность управления

- **Вариант Б:** Использование прокси для одного бота
  - Минусы: Не решает проблему блокировки аккаунта, только IP

- **Вариант В:** Резервный юзербот (standby)
  - Минусы: Не используется в нормальном режиме, переключение требует времени

- **Выбран:** Параллельные юзерботы с равномерным распределением
  - Почему: Максимальная отказоустойчивость, параллельная обработка, автоматическое переключение

## Распределение каналов

### Стратегия: Round-Robin с учетом нагрузки

1. При добавлении канала выбирается бот с наименьшим количеством каналов
2. При равной нагрузке используется round-robin
3. При блокировке бота его каналы перераспределяются равномерно

### Пример распределения:

- **Userbot1:** Каналы 1, 4, 7, 10 (4 канала)
- **Userbot2:** Каналы 2, 5, 8 (3 канала)
- **Userbot3:** Каналы 3, 6, 9 (3 канала)
