# Паттерны и Стандарты кода

## Общие правила

- Все настройки (токены, пароли) выносятся в `.env` и загружаются через `data/config.py`.
- Используется **SQLite** через стандартную библиотеку `sqlite3`. Важно использовать параметризованные запросы (`?`), чтобы избежать SQL-инъекций.
- Логирование через модуль `logging`. `print()` не используется для продакшн-логов.
- Все вызовы API Telegram и БД оборачиваются в блоки `try-except`.

## Обработка ошибок

- Все вызовы API Telegram и БД оборачиваются в блоки `try-except`.
- Ошибки логируются с уровнем `ERROR` или `CRITICAL`.
- При ошибке отправки сообщения пользователю, работа бота не должна прерываться.
- Для обработки ошибок Telegram API (RetryAfter) используется задержка через `asyncio.sleep()`.

## Структура проекта

Проект разделен на модули для разделения ответственности:
- `handlers/` - обработчики сообщений
- `database/` - работа с БД
- `utils/` - утилиты и бизнес-логика
- `keyboards/` - клавиатуры

## Структура хендлеров

- Используются декораторы `@dp.message_handler` и `@dp.callback_query_handler`.
- Хендлеры разделены по смыслу (админские, пользовательские, авторские).
- Для многошаговых диалогов используется FSM через `utils/states_users.py`.

## Безопасность

### Секреты и конфиденциальные данные

- **НИКОГДА** не хардкодить пароли, токены, API-ключи в коде
- **НИКОГДА** не писать секреты в чате с агентом (даже в примерах)
- **НИКОГДА** не добавлять секреты в логи
- Все секреты хранить в `.env` (который в `.gitignore`)
- Файл `.env` всегда должен быть в `.gitignore`

### Валидация входных данных

- Все данные от пользователей должны валидироваться перед записью в БД
- Использовать параметризованные SQL-запросы (всегда `?`, никогда не форматировать строки)
- Проверять типы данных и диапазоны значений
- Использовать `validators.py` для валидации

### SQL-инъекции

- **КРИТИЧЕСКИ ВАЖНО:** Не использовать f-строки в SQL-запросах с пользовательскими данными
- Использовать параметризованные запросы: `cursor.execute(query, params)`
- Для динамических имен столбцов использовать валидацию и whitelist

## Логирование

### Правила логирования

- **Логировать все важные операции:**
  - Действия администратора (изменение подписок, добавление каналов)
  - Ошибки подключения к БД
  - Ошибки API Telegram
  - Операции с подписками (создание, продление, истечение)
  - Рассылки и экспорты
  - Ошибки при отправке сообщений

### Уровни логирования

- `DEBUG` - детальная информация для отладки
- `INFO` - общая информация о работе бота
- `WARNING` - предупреждения (некритичные проблемы)
- `ERROR` - ошибки, которые не прерывают работу
- `CRITICAL` - критические ошибки (БД недоступна, бот не может запуститься)

### Формат логов

```python
logger.info(f"Подписка пользователя {user_id} продлена до {end_date}")
logger.error(f"Ошибка при удалении пользователя {user_id}: {e}")
logger.warning(f"Пользователь {user_id} не найден в группе")
```

**Важно:** В логах не должно быть секретов (токены, пароли, API-ключи).

## Тестирование

### Обязательные правила

- **На каждую функцию пишутся тесты**
- Тесты планируются еще на этапе ТЗ (в spec-template.md)
- Тесты запускаются после каждой задачи
- Тесты должны проходить перед коммитом

### Структура тестов

- Тесты хранятся в папке `tests/`
- Имя файла: `test_<имя_модуля>.py`
- Использовать библиотеку `pytest`

## Работа с БД

### Правила

- Использовать функции из `database/getters.py` и `database/setters.py`
- Не создавать прямые SQL-запросы в handlers
- Все запросы должны быть параметризованными
- Использовать транзакции для критических операций

### Примеры

```python
# Правильно
user_data = await get_data("users", user_id=user_id)
await add_new_data("user_subscriptions", data=(user_id, end_date))

# Неправильно
query = f"SELECT * FROM users WHERE user_id = {user_id}"  # SQL-инъекция!
```

## Работа с FSM

- Использовать функции из `utils/states_users.py`
- Всегда сбрасывать состояние после завершения диалога
- Сохранять данные в state для многошаговых процессов

## Self-Review (Самопроверка агента)

### Перед началом работы

1. Оценить план решения:
   - Это лучшее решение? Почему?
   - Какие подводные камни?
   - Какие риски?
   - Альтернативные решения?

2. Для каждой задачи определить:
   - Критерии успеха
   - Чеклист самопроверки

3. **КРИТИЧЕСКИ ВАЖНО: Проверить связанные функции**
   - Перед изменением любого обработчика/функции найти все места, где она используется
   - Проверить, какие другие функции могут быть затронуты
   - Убедиться, что изменения не сломают существующую функциональность
   - Использовать `grep` или `codebase_search` для поиска всех использований

### После выполнения задачи

1. Пройтись по чеклисту из спека
2. **Проверить все связанные функции:**
   - Убедиться, что все обработчики кнопок/команд работают
   - Проверить, что меню отображаются корректно
   - Убедиться, что навигация между меню работает
   - Проверить, что не сломались существующие функции
3. Провести аудит кода:
   - Потенциальные проблемы и уязвимости
   - Неиспользуемые функции
   - Несоответствия patterns.md
   - Проблемы производительности
   - Проблемы безопасности

4. Если найдены проблемы - исправить и обновить документацию

### Правила тестирования перед коммитом

**ОБЯЗАТЕЛЬНО перед каждым коммитом:**
1. Проверить все обработчики, связанные с изменяемым кодом
2. Убедиться, что все кнопки в меню имеют обработчики
3. Проверить, что навигация (кнопки "Назад", "Главное меню") работает
4. Убедиться, что не сломались существующие функции
5. Проверить логику работы с базой данных (если изменялась)
6. Убедиться, что все импорты корректны

**Если что-то сломалось:**
- Немедленно исправить
- Не коммитить сломанный код
- Проверить все связанные функции после исправления
