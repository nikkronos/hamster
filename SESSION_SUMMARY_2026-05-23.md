## Архитектурные решения

1. **Единая БД для campaign-данных** — все `broadcasts` и `broadcast_clicks` живут в БД hamster26 (`/home/hamster26/bot/database/database.sqlite`). Hamster93 пишет туда через прямой SQLite-коннект (absolute path). Cross-DB SQLite OK поскольку оба бота на одном хосте.

2. **Deep link через инвертирующий бот** — рассылки шлются из hamster93 (где основная база), но кнопка ведёт в hamster26 (где активный продукт). Telegram автоматически открывает целевого бота.

3. **Промокод не виден пользователю** — применяется только через FSM-state, проставленный при клике на кнопку. Утечки кода исключены.

4. **Списки исключений в файлах** — `data/broadcast_excludes.txt`, по одному user_id на строку. Дублируется в обоих ботах.

5. **Тестовый аккаунт hardcoded** — `TEST_USER_ID = 7882817110` в broadcast.py. Принадлежит Anton tosha8686, проверено что он зарегистрирован в обоих ботах.

## Известные ограничения

1. **State persistence:** aiogram использует MemoryStorage. При рестарте бота FSM-state теряется. Если юзер кликнул на кнопку, но не оплатил до рестарта — придётся кликать снова. **Митигация:** не рестартить боты во время кампании.

2. **Orphan clicks:** при отмене рассылки удаляется row из `broadcasts`, но соответствующие `broadcast_clicks` остаются. Stats query через JOIN их не показывает, так что для UI это не проблема. При желании можно почистить через `DELETE FROM broadcast_clicks WHERE broadcast_id NOT IN (SELECT id FROM broadcasts)`.

3. **Промокод единый, не персональный.** Если кто-то узнает что код `INDIVIDUAL2026` существует, теоретически может ввести его руками. Но он короткий срок действия + мы его не показываем.

4. **Цена 1000₽ захардкожена** в `start.py` (`DISCOUNT_PRICE = 1000`). Если будет другой промокод с другой ценой — нужно пересмотреть. Сейчас 33% от 1500 = 1005, но мы override-им до ровного 1000.
